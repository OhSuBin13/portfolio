# Multi-Broker Integration Strategy

작성일: 2026-06-19

이 문서는 토스증권과 나무증권/NH투자증권을 모두 포트폴리오 앱에
연결하는 전략을 정리한다. 세부 API 조사 내용은 다음 문서를 기준으로
한다.

- `docs/toss-open-api-integration.md`
- `docs/namuh-open-api-integration.md`

## 1. 결론

장기적으로는 여러 증권사를 연결할 수 있는 구조가 맞다. 이 앱은 특정
증권사 앱을 대체하는 거래 앱이 아니라, 여러 계좌와 자산을 모아 보는
개인 포트폴리오 원장에 가깝기 때문이다.

다만 지금 바로 토스증권과 나무증권을 같은 수준으로 붙이는 것은
권장하지 않는다. 토스증권은 공식 REST API 문서와 기능 범위가 확인된
반면, 나무증권/NH투자증권은 공개적으로 검증 가능한 REST API 레퍼런스가
확인되지 않았고, 공식 페이지상으로는 N2/QV 연결 모듈 기반의 트레이딩
채널 성격이 강하다.

따라서 권장 방향은 다음과 같다.

1. 내부 구조는 다중 증권사 지원을 전제로 설계한다.
2. 첫 구현은 토스증권 read-only 연동으로 시작한다.
3. 나무증권은 공식 개발 문서나 연결 모듈 매뉴얼 확보 전까지 보류한다.
4. 직접 주문 기능은 두 증권사 모두 후순위로 둔다.

## 2. 원칙

### 2.1 로컬 SQLite가 계속 원본이다

증권사 API를 붙여도 `accounts`, `assets`, `holdings`, `transactions`,
`price_snapshots`, `fx_rates`, `portfolio_snapshots`는 앱의 원본 데이터
모델로 유지해야 한다.

증권사 데이터는 다음 역할만 맡는다.

- 시세와 환율을 가져와 로컬 스냅샷으로 저장한다.
- 외부 계좌와 보유자산을 가져와 로컬 데이터와 비교한다.
- 체결 내역을 가져와 로컬 거래내역 후보로 제안한다.
- 사용자가 확인한 뒤에만 로컬 원장에 반영한다.

증권사 API 응답이 로컬 원장을 자동으로 덮어쓰는 구조는 피해야 한다.
API 장애, 누락, 권한 제한, 상품 미지원 때문에 로컬 데이터가 사라질 수
있기 때문이다.

### 2.2 Read-only를 먼저 구현한다

첫 연결 범위는 조회와 동기화에 한정한다.

- 시장 시세
- USD/KRW 환율
- 종목 메타데이터
- 증권사 계좌 목록
- 보유자산 조회
- 체결내역 조회

주문 생성, 정정, 취소는 별도 제품 영역으로 취급한다. 직접 주문은
사용자의 실제 자산에 영향을 주므로 주문 전 확인, 예수금, 매도 가능
수량, 시장 상태, 가격 제한, 오류 복구, 주문 후 체결내역 반영까지 별도
설계가 필요하다.

### 2.3 증권사별 기능 차이를 숨기지 않는다

모든 증권사를 같은 기능으로 추상화하려고 하면 오히려 위험하다. 각
증권사는 API 제공 범위, 상품 범위, 인증 방식, 계좌 권한, 주문 제한,
오류 형식이 다를 수 있다.

따라서 내부 provider 인터페이스는 공통 기능을 제공하되, 실제 화면과
서비스는 provider capability를 확인해 가능한 기능만 노출해야 한다.

예시 capability:

- `market_prices`
- `fx_rates`
- `stock_metadata`
- `accounts`
- `holdings`
- `order_history`
- `order_placement`
- `requires_local_bridge`
- `supports_rest`

## 3. 권장 구현 순서

### Phase 1. Provider 경계 만들기

첫 단계에서 토스 전용 코드를 바로 서비스 내부에 흩뿌리면 나중에
나무증권을 붙일 때 구조를 다시 뜯게 된다. 먼저 provider 경계를 만든다.

권장 경계:

- `MarketDataProvider`: 종목별 현재가와 통화 정보를 반환한다.
- `FxRateProvider`: USD/KRW 같은 환율을 반환한다.
- `BrokerAccountProvider`: 외부 계좌 목록을 반환한다.
- `BrokerHoldingProvider`: 외부 보유자산을 반환한다.
- `BrokerOrderHistoryProvider`: 체결 또는 주문 이력을 반환한다.

첫 구현체는 토스만 있어도 된다. 중요한 것은 서비스 계층이 토스 API
형식을 직접 알지 않도록 하는 것이다.

### Phase 2. 토스증권 시세와 환율 연동

토스증권은 현재 가장 낮은 비용으로 붙일 수 있는 후보이다.

먼저 다음 기능을 구현한다.

- KR/US 주식/ETF 현재가 조회
- USD/KRW 환율 조회
- `price_snapshots` 저장
- `fx_rates` 저장
- 기존 stale price fallback 유지
- 실패 시 로컬 요약 계산을 깨뜨리지 않는 오류 처리

이 단계는 현재 앱의 직접적인 기능 격차를 줄인다. 특히 KR 시장 시세
동기화가 현재 미지원이므로 효과가 크다.

### Phase 3. 토스증권 종목정보와 보유자산 preview

다음 단계는 로컬 원장 반영 전 preview이다.

권장 흐름:

1. 토스 계좌 목록 조회
2. 사용자가 연결할 계좌 선택
3. 토스 보유자산 조회
4. 로컬 `assets`/`holdings`와 차이 비교
5. 추가/변경 후보를 테이블로 표시
6. 사용자가 확인한 항목만 반영

이 단계에서도 자동 삭제는 하지 않는다. 토스에 없는 자산은 비토스
자산, 미지원 상품, 수동 입력 자산일 수 있다.

### Phase 4. 토스증권 체결내역 import preview

체결내역은 수동 buy/sell 입력을 줄일 수 있지만, 현재 거래 모델 전체를
대체하지는 않는다.

필요한 설계:

- 증권사별 외부 주문 ID 또는 deduplication key 저장
- 수수료와 세금 표현 방식 결정
- 부분 체결 처리
- 배당, 이자, 입출금, 조정 거래와의 경계 유지
- import 실행 전 preview와 사용자 확인

이 단계까지도 read-only 기반이다. 외부 체결내역을 읽어서 로컬 거래
후보를 만드는 것이지, 앱에서 주문을 내는 것이 아니다.

### Phase 5. 나무증권/NH투자증권 재평가

나무증권은 다음 자료가 확보된 뒤에만 구현을 검토한다.

- 공식 개발자 가이드
- API 레퍼런스 또는 연결 모듈 매뉴얼
- 인증과 계좌 권한 문서
- 운영/실전 사용 승인 조건
- 시세, 잔고, 주문, 체결 schema
- 오류 코드와 rate limit
- 지원 OS와 설치 요구사항
- 테스트 환경 또는 모의투자 환경

만약 REST API가 아니라 연결 모듈 기반이라면 일반 provider가 아니라
bridge provider로 분리해야 한다.

예상 구조:

- FastAPI backend는 그대로 유지한다.
- Windows 또는 지원 환경에서 별도 bridge process를 실행한다.
- bridge가 나무/N2 연결 모듈과 통신한다.
- backend는 bridge의 제한된 local API만 호출한다.
- 처음에는 보유자산/시세 같은 read-only 기능만 허용한다.

이 구조는 토스 REST 연동보다 복잡하므로 1차 구현 대상으로 두지 않는다.

## 4. 데이터 모델 방향

다중 증권사를 붙이려면 로컬 ID와 외부 ID를 분리해야 한다.

권장 개념:

- 로컬 `accounts.id`는 앱 내부 계좌 ID이다.
- 외부 증권사 계좌 ID는 provider namespace와 함께 별도로 저장한다.
- 로컬 `assets.id`는 앱 내부 자산 ID이다.
- 외부 종목 코드와 거래소 정보는 매핑 테이블이나 별도 필드로 관리한다.
- 외부 주문 ID는 provider와 함께 저장해 중복 import를 막는다.

초기 구현에서는 큰 schema 변경 없이 provider 결과를 preview로 보여줄
수 있다. 실제 반영 기능을 넣을 때는 다음 개념이 필요해질 가능성이 높다.

- `broker_connections`: 연결된 증권사와 상태
- `broker_accounts`: 외부 계좌와 로컬 계좌의 매핑
- `broker_asset_mappings`: 외부 종목과 로컬 자산의 매핑
- `broker_import_runs`: 가져오기 실행 기록
- `broker_external_transactions`: 외부 주문/체결 deduplication 기록

이름과 schema는 실제 구현 시 현재 DB 구조에 맞춰 다시 설계한다.

## 5. 화면 흐름 방향

사용자에게는 증권사 연결을 설정 화면 또는 별도 "연동" 화면에서
관리하게 하는 편이 좋다.

권장 화면:

- 연결된 증권사 목록
- 증권사별 연결 상태
- 마지막 동기화 시각
- 사용 가능한 기능
- 계좌 매핑 상태
- 동기화 오류
- 보유자산 preview
- 체결내역 import preview

보유자산 화면이나 거래내역 화면에서는 외부 데이터를 바로 쓰는 대신
비교 결과와 반영 후보를 보여준다.

## 6. 보안 원칙

증권사 연동은 API key나 인증 토큰이 직접 자산 정보와 거래 권한에
연결될 수 있으므로 보수적으로 다뤄야 한다.

원칙:

- 증권사 secret은 frontend로 보내지 않는다.
- 설정은 `PORTFOLIO_` prefix를 쓰는 backend 설정으로 관리한다.
- 토큰 저장이 필요하면 별도 암호화 저장 설계 전까지 파일/DB 저장을
  피한다.
- 주문 권한 scope는 read-only 안정화 전까지 요청하지 않는다.
- provider 오류 메시지는 사용자에게 필요한 수준으로 정제한다.
- 테스트에서는 live API를 호출하지 않고 mock response를 사용한다.

## 7. 직접 주문 기능에 대한 입장

토스증권과 나무증권을 모두 연결하더라도 직접 주문은 첫 목표가 아니다.

직접 주문을 넣으려면 다음이 필요하다.

- 주문 전 preview
- 사용자 명시 확인
- 예수금/매수 가능 금액 확인
- 매도 가능 수량 확인
- 시장 개장 여부 확인
- 가격 제한과 호가 단위 확인
- 주문 실패와 중복 주문 방지
- 주문 정정/취소 흐름
- 체결 후 로컬 거래내역 반영
- 감사 로그

이 기능은 포트폴리오 트래킹 앱을 트레이딩 시스템으로 바꾸는 변화이므로
read-only 동기화가 충분히 안정화된 뒤 별도 설계 문서와 테스트 계획을
두고 진행해야 한다.

## 8. 최종 권고

다중 증권사 지원은 제품 방향과 잘 맞는다. 하지만 구현 순서는 다중
연결을 바로 완성하는 방식이 아니라, 다중 provider가 가능한 구조를 먼저
잡고 토스증권으로 검증하는 방식이 적절하다.

권장 결정:

- 지금은 토스증권을 1차 provider로 둔다.
- 나무증권은 공식 문서 확보 전까지 구현하지 않는다.
- 내부 모델은 처음부터 provider 확장을 고려한다.
- 증권사 데이터는 로컬 원장의 입력원으로만 사용한다.
- 모든 보유자산/체결내역 반영은 preview-confirm 방식으로 처리한다.
- 주문 기능은 별도 프로젝트로 분리한다.

이 접근이 가장 현실적인 이유는 토스 연동으로 즉시 해결 가능한 기능
격차가 크고, 동시에 나무증권을 나중에 붙일 수 있는 구조적 여지를
남길 수 있기 때문이다.
